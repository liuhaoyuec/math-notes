# Python OOP 底层内存与继承机制 (反黑盒版)

## 1. 核心世界观：图纸与机器

在 Python 的底层内存模型中，类（Class）和实例（Instance）有着极其严格的物理隔离。

* **类 (Class) = 基因图纸**：只保存在内存的代码区。**所有的函数（方法）统统存放在这里**。无论造多少台机器，图纸永远只有一份。（为了节省内存）
* **实例 (Instance) = 物理机器**：真正开辟在堆区的实体。这台机器的肚子里只有一个扁平的字典 `__dict__`。**所有的变量统统存放在这里**。机器里绝对不存任何函数代码。

> [!abstract] 终极定律
> **函数归图纸，变量归机器。**
> 调用函数靠指针爬树，生成变量直接塞进机器肚子。

---

## 2. 继承的本质：指针回溯 (Pointer Chaining)

继承绝对不是复制粘贴代码！

当你写下 `class CNN_Engine(Machine):` 时，子类 `CNN_Engine` 的图纸上并没有父类的代码，而是悄悄打上了一个内存指针（`__bases__`），死死地指向了 `Machine` 图纸的内存地址。

### 方法解析顺序 (MRO - 找函数的逻辑)
当你执行 `engine.run()` 时，底层发生了“三段式跳跃寻址”：
1.  **查实例肚子**：看 `engine.__dict__` 里有没有叫 `run` 的东西。
2.  **查子类图纸**：如果没有，触发隐形指针 `__class__`，去 `CNN_Engine` 图纸的技能表里找。
3.  **查父类图纸**：如果图纸上也没有，顺着继承指针去爹 `Machine` 那里白嫖。

## 3. `__init__` 与变量生命周期

> [!question] 疑问 1：子类对象化后，会自动执行父类的 `__init__` 吗？
> **绝对不会！** > 只要子类自己写了 `def __init__(self):`，父类的初始化就被“物理屏蔽”了。

为了让父类也能给这台机器塞入基础变量，必须手动拉网线——使用 `super()`。

```python
class Machine:
    def __init__(self):
        self.core_voltage = 220  # 父类往机器里塞入变量

class CNN_Engine(Machine):
    def __init__(self):
        super().__init__() # 【极其关键】把当前机器交给父类，让爹先塞变量！
        self.gpu_count = 4   # 爹干完活了，子类接着往同一个肚子里塞
