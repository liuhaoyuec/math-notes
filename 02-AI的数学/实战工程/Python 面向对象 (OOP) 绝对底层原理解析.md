# 🧠 Python 面向对象 (OOP) 绝对底层原理解析

> [!abstract] 核心哲学：一切皆对象，对象皆字典
> 在 Python 的底层，**类 (Class)** 只是一个装满函数指针的“共享代码库”；而**实例 (Instance)** 只是一个装满数据的“肉身字典”。面向对象编程，本质上就是把“肉身字典”当做参数，传给“共享代码库”里的函数。

---

## 1. 终极造物悖论：对象到底是怎么生出来的？

> **💡 破除误区**：`__init__` 根本不是用来“创造”对象的！如果对象没创造出来，它去哪里拿 `self`？

对象的诞生是一个绝对的**两步走战略**：

1.  **第一步：`__new__` (真正的造物主/构造器)**
    * **物理动作**：向操作系统的堆内存 (Heap) 申请一块空间，开辟出一个绝对干净的“空壳字典”。
    * *此时，实体在物理世界已经真实存在，悖论解除！*
2.  **第二步：`__init__` (精装修包工头/初始化器)**
    * **物理动作**：系统将 `__new__` 造出来的空壳的内存地址，强行塞进 `__init__` 的第一个参数 `self` 中。
    * `__init__` 拿到这个空壳，开始往里面塞数据：`self.name = "亚瑟"`，`self.hp = 100`。

*(注：即使完全不写 `__init__`，系统依然会用 `__new__` 给你造一个合法的空壳字典。)*

---

## 2. 空间隔离法则：函数在天，数据在地

> **💡 破除误区**：当你 `w1 = Warrior()` 时，系统绝对**没有**把图纸里的函数复制一份塞进 `w1` 肚子里！

* **数据 (Data)**：跟随着实体 `w1`，生存在堆内存里。每一个战士都有自己独立的肉身字典 `w1.__dict__`。
* **函数 (Methods)**：永远只有唯一的一份，挂载在高高在上的图纸 (Class) 里。
* **连接的脐带 (`__class__`)**：`w1` 字典的底部藏着一根指针，指向天上的图纸。当 `w1` 需要执行动作时，它会顺着脐带去天上找代码。

---

## 3. `self` 的绝对物理意义

因为函数在天上只有一份，当全天下几万个战士同时调用 `attack()` 时，函数必须知道在扣谁的血。

* **语法糖**：`w1.attack()` 
* **底层真面目**：`Warrior.attack(w1)`
* **本质**：编译器偷偷把你调用的对象 `w1` 的内存地址，作为第一个参数（也就是 `self`）传给了天上的函数。`self` 就是实体的**唯一内存身份证**。

---

## 4. 路由分歧：带 `__` 与不带 `__` 的执行鸿沟

它们在 CPU 和内存里的执行路径截然不同：

> [!info] 类别 A：魔术方法 (如 `__init__`, `__len__`) —— 【C 语言槽位直达】
> * **触发**：人类写出极其优美的语法糖（如 `len(w1)` 或 `w1 = Warrior()` 时自动触发）。
> * **底层机制**：拦截语法树！绕过 Python 字典查找，直接通过指针硬件级直达 CPython 底层的物理槽位 (Slots)。极速执行，系统拥有绝对强制力。

> [!warning] 类别 B：普通函数 (如 `attack`) —— 【动态字典搜索】
> * **触发**：必须由人类主动手动调用（如 `w1.attack()`）。
> * **底层机制**：执行极其繁琐的两次哈希搜索。先去 `w1` 的肉身字典找（没找到），再顺着脐带去类字典找（找到了），然后临时生成绑定方法 (Bound Method) 压栈执行。

---

## 5. 危险的动态自由：薛定谔的属性

> **💡 破除误区**：普通的函数到底能不能创建 `self.weapon = "剑"`？能！但极其危险。

* **Python 的野蛮字典**：因为对象底层就是个字典，所以你随时随地，在任何普通函数里执行 `self.属性 = 值`，都能强行向字典里插值。
* **工程死穴**：如果不在 `__init__` 里把所有坑位挂载好，而是把创建属性的代码写在普通函数里，那么在普通函数被人类调用之前，这个属性**根本不存在**！此时任何人尝试读取它，都会直接引发 `AttributeError` 导致内存崩溃空指针。

**🔥 架构师铁律**：
所有的基础数据贴牌（定义属性），**必须、绝对、只能**写在 `__init__` 里！普通函数只允许修改它们，绝不允许无中生有地创造它们。
