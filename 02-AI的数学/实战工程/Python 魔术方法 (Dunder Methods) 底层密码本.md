# Python 魔术方法 (Dunder Methods) 底层密码本

## 1. 核心世界观：被动触发机制

在 Python 中，所有带有双下划线的方法（如 `__len__`, `__init__`）被称为 **Dunder Methods (Double Underscore)** 或 **魔术方法**。

> [!info] 核心定律：它们是“被动技能”
> Python 官方设计这些方法的初衷，**绝对不是让人类手动去调用的**。它们是留给系统语法的“底层钩子（Hook）”。
> 永远让系统去触发它们，绝不要自己写 `engine.__len__()`。

---

## 2. 裸奔调用 vs 带安检调用

> [!danger] 为什么 `len(engine)` 和 `engine.__len__()` 绝对不等价？
> 
> * **`engine.__len__()` (裸奔调用)**：绕过所有系统检查，强行执行内部代码。如果返回了乱码字符串，系统也会照单全收，导致后续深度学习运算彻底崩溃。
> * **`len(engine)` (带安检调用)**：C 语言底层的质检员接管。它会暗中触发 `__len__`，并**强行拦截校验**返回值（必须是 >=0 的整数），一旦发现异常，当场抛出 `TypeError`，保护程序不被脏数据摧毁。

---

## 3. 为什么不能写个 for 循环一口气全执行？

1. **参数契约不同**：`__len__` 不要参数，`__getitem__` 必须要索引参数，盲目批量执行会瞬间触发 `TypeError` 爆炸。
2. **物理破坏性**：包含 `__del__` 这种内存回收方法，强行执行会直接把对象在内存里“扬了灰”。

---

## 4. 核心魔术方法与触发语法速查表

> [!abstract] 深度学习高频魔术方法清单

### ⚙️ 一、 生命周期（造机器与毁机器）
* `__init__(self, ...)` ➡️ **触发：** `engine = ClassName(...)` (装配内部变量)
* `__new__(cls, ...)` ➡️ **触发：** 极少使用，真正向系统申请物理内存的动作。
* `__del__(self)` ➡️ **触发：** `del engine` (临终遗言，销毁内存)

### 📦 二、 容器与数据集（Dataset 核心）
* `__len__(self)` ➡️ **触发：** `len(engine)` (返回数据总量)
* `__getitem__(self, key)` ➡️ **触发：** `engine[key]` (按索引或键精确提取数据)
* `__setitem__(self, key, val)` ➡️ **触发：** `engine[key] = val` (修改容器内数据)
* `__iter__(self)` ➡️ **触发：** `for item in engine:` (支持被循环遍历)

### 🗣️ 三、 表达与打印
* `__str__(self)` ➡️ **触发：** `print(engine)` (给人看的易读字符串)
* `__repr__(self)` ➡️ **触发：** 控制台回车 / `repr(engine)` (给机器看的严谨字符串，用于 Debug)

### ➕ 四、 数学与张量运算（微积分必备）
* `__add__(self, other)` ➡️ **触发：** `A + B` (定义张量相加)
* `__sub__(self, other)` ➡️ **触发：** `A - B`
* `__mul__(self, other)` ➡️ **触发：** `A * B`
* `__matmul__(self, other)` ➡️ **触发：** `A @ B` (深度学习极其高频的**矩阵乘法**)

### 🚀 五、 终极黑魔法
* `__call__(self, *args)` 
	* ➡️ **触发：** `engine(图片)`
	* ➡️ **底层：** 把物理机器直接当成函数执行！PyTorch 的前向传播 `output = model(image)` 100% 依赖它！
* `__getattr__(self, name)` 
	* ➡️ **触发：** `engine.不存在的变量`
	* ➡️ **底层：** 机器肚子里和图纸上都找不到变量时，系统给的最后一次抢救机会。
